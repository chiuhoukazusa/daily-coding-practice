# Daily Coding Practice Report Skill

## 适用场景
每日编程实践完成后，自动生成一份结构化的日报，总结当天的学习成果、遇到的问题、解决方案和技术总结。

## 核心原则

**日报不是流水账**：
- 不是简单罗列做了什么
- 重点是**学到了什么**、**遇到了什么问题**、**如何解决的**
- 技术总结可复用（作为未来的参考）

**日报是记忆的延伸**：
- 每天的 bug 和解决方案要记录
- 关键决策和理由要记录
- 下次遇到类似问题可以快速查找

## 自动化生成

### 方式1：手动填写

每天编程结束后，按模板填写日报，存放到：
```
/root/.openclaw/workspace/daily-coding-practice/reports/YYYY-MM-DD.md
```

### 方式2：半自动生成

```bash
#!/bin/bash
# generate_daily_report.sh

DATE=$(date +%Y-%m-%d)
PROJECT_DIR=$(ls -td 2026/02/* | head -1)
PROJECT_NAME=$(basename "$PROJECT_DIR")

# 提取信息
CODE_LINES=$(find "$PROJECT_DIR" -name "*.cpp" -o -name "*.py" -o -name "*.js" | xargs wc -l | tail -1 | awk '{print $1}')
COMMIT_HASH=$(git log --oneline -1 "$PROJECT_DIR" | awk '{print $1}')
START_TIME=$(git log --format=%ai "$PROJECT_DIR" | tail -1 | awk '{print $2}')
END_TIME=$(git log --format=%ai "$PROJECT_DIR" | head -1 | awk '{print $2}')

# 生成报告
cat > reports/$DATE.md << EOF
# 每日编程实践 - $DATE

## 📦 项目概览
- **项目名称**：$PROJECT_NAME
- **完成时间**：$START_TIME - $END_TIME
- **代码行数**：$CODE_LINES 行
- **提交记录**：commit \`$COMMIT_HASH\`

## 🎯 核心功能
（请手动填写）

## 🐛 遇到的问题
（请手动填写）

## 📚 技术总结
（请手动填写）

## 🔗 产出
- 代码仓库：https://github.com/chiuhoukazusa/daily-coding-practice/tree/main/$PROJECT_DIR
- 博客文章：（请手动填写）

## 🚀 下一步
（请手动填写）
EOF

echo "✅ 日报框架已生成：reports/$DATE.md"
echo "请手动补充核心内容"
```

## 使用流程

### 每日流程
1. **完成编程实践**
2. **立刻生成日报框架**（趁记忆新鲜）
   ```bash
   bash generate_daily_report.sh
   ```
3. **填写核心内容**（15-30分钟）
   - 遇到的问题和解决方案
   - 技术总结
   - 学到的教训
4. **提交到仓库**
   ```bash
   git add reports/
   git commit -m "Add: 每日报告 - YYYY-MM-DD"
   git push
   ```

### 每周回顾
- 阅读本周的所有日报
- 总结共性问题和解决方案
- 提取技术要点到 `MEMORY.md`

### 每月总结
- 统计本月完成的项目数量
- 总结技术成长路径
- 规划下月学习方向

## 关键原则

### 1. 及时记录
**在编程结束后立刻写日报**，不要拖到第二天。记忆新鲜时记录的细节更准确。

### 2. 重点突出
不要流水账式记录所有操作，重点记录：
- **遇到的问题**（最有价值）
- **解决方案**（可复用）
- **技术总结**（知识沉淀）

### 3. 可检索
使用统一的格式和关键词，方便未来搜索。例如：
- 问题标题：`## 问题：图像上下颠倒`
- 技术点：`### 学到：球面UV映射`

### 4. 量化优于定性
- ❌ "渲染很快"
- ✅ "渲染时间 1.2 秒（800x600）"

### 5. 失败也要记录
失败的尝试和错误的分析同样有价值，记录下来避免重复犯错。

## 与其他工具的集成

### 与 MEMORY.md 的关系
- **日报**：每日的原始记录（流水）
- **MEMORY.md**：提炼的长期记忆（精华）

定期将日报中的重要内容提炼到 MEMORY.md。

### 与博客的关系
- **日报**：完整的技术过程（包含bug和解决方案）
- **博客**：对外展示的成果（只展示最终结果）

日报可以作为博客的素材来源。

### 与 GitHub Commit 的关系
- **Commit message**：简短的修改说明（一句话）
- **日报**：详细的上下文和原因（完整故事）

日报可以引用 commit hash，方便回溯代码变更。

## 常见问题

### Q1: 每天写日报太花时间怎么办？
**A**: 使用半自动生成框架，只手动填写核心内容（问题+解决方案+技术总结），15-30分钟即可。

### Q2: 日报和博客有什么区别？
**A**: 
- 日报：私人记录，包含所有 bug 和失败尝试
- 博客：公开展示，只展示最终成果和精华

### Q3: 日报要写多详细？
**A**: 详细到**6个月后的自己能看懂并复现**。

### Q4: 日报要保存多久？
**A**: 永久保存。这是你的技术成长档案。
